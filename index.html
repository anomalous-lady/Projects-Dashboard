What is Streamlit and why did I use it?

Streamlit is a Python framework that turns a plain .py script into a full web app — with UI widgets, charts, file uploaders — without writing any HTML, CSS, or JavaScript. You write Python top-to-bottom and Streamlit re-runs the whole script every time the user changes something.

The mental model: Think of a Streamlit app as a Python script that auto-reruns from the top whenever any widget changes. That's literally all it is. Every st.slider(), st.selectbox(), or st.button() returns a value you use in normal Python code.

I used Streamlit here instead of React/FastAPI because for data apps, the whole point is the data and charts, not the UI. Streamlit gets you to a deployable live demo in 30 minutes. That's perfect for a freelance portfolio piece.

What is yfinance and how does data flow in?

yfinance is a Python library that wraps Yahoo Finance's API. You give it a ticker symbol and a period, and it gives you back a pandas DataFrame with OHLCV data (Open, High, Low, Close, Volume).

Pythondata_loading.py
import yfinance as yf
import pandas as pd

stock = yf.Ticker("AAPL")

df = stock.history(period="3mo")

print(df.head())

info = stock.info
print(info['longName'])
print(info['marketCap'])
print(info['sector'])
Why cache it? yfinance makes a real HTTP request to Yahoo's servers every call. Calling it on every Streamlit rerun would be slow and could get rate-limited. We use @st.cache_data(ttl=300) to cache the result for 5 minutes — so the same ticker only fetches once per 5 minutes no matter how many times the user interacts with the app.
Pythoncaching_pattern.py
import streamlit as st

@st.cache_data(ttl=300)
def load_data(ticker, period):
    stock = yf.Ticker(ticker)
    df = stock.history(period=period)
    info = stock.info
    return df, info

df, info = load_data("AAPL", "3mo")

Computing financial metrics from scratch

This is where pandas and numpy do the heavy lifting. Every metric is just math on the df['Close'] column.

Pythonmetrics.py
import numpy as np

daily_returns = df['Close'].pct_change()

volatility = daily_returns.std() * np.sqrt(252) * 100

risk_free_rate = 0.04
sharpe = (daily_returns.mean() * 252 - risk_free_rate) / \
         (daily_returns.std() * np.sqrt(252))

rolling_max = df['Close'].cummax()
drawdown = (df['Close'] - rolling_max) / rolling_max
max_drawdown = drawdown.min() * 100

delta = df['Close'].diff()
gain = delta.clip(lower=0).rolling(14).mean()
loss = (-delta.clip(upper=0)).rolling(14).mean()
rs = gain / loss
rsi = 100 - (100 / (1 + rs))

How Isolation Forest works — the core idea

Most anomaly detection asks "is this point far from the cluster?" Isolation Forest asks something cleverer: "how easy is it to isolate this point?"

Imagine a forest of random decision trees. Each tree randomly picks a feature and a split value, over and over, until each data point is in its own leaf. Anomalies get isolated in very few splits. Normal points buried inside a dense cluster take many splits to isolate.

Intuition: If I asked you to find a needle in a haystack vs. a haystack in a haystack — the needle gets found (isolated) very quickly. Anomalies = needles. Normal points = haystacks.
Pythonisolation_forest.py
from sklearn.ensemble import IsolationForest
from sklearn.preprocessing import StandardScaler
import numpy as np

scaler = StandardScaler()
X_scaled = scaler.fit_transform(df[numeric_cols])

model = IsolationForest(
    n_estimators=100,
    contamination=0.05,
    max_features=0.8,
    random_state=42
)
model.fit(X_scaled)

labels = model.predict(X_scaled)
scores = model.score_samples(X_scaled)

df['is_anomaly'] = labels == -1
df['anomaly_score'] = scores

n_anomalies = (labels == -1).sum()
print(ff"Found {n_anomalies} anomalies out of {len(labels)} rows")

PCA — Visualising high-dimensional data in 2D

Your data might have 10 features. You can't plot 10 dimensions. PCA (Principal Component Analysis) finds the 2 directions in your data that capture the most variation, and projects everything onto those 2 axes.

Think of it like a shadow: If you have a 3D sculpture and shine a light at the perfect angle, the shadow captures most of the sculpture's shape in 2D. PCA finds that "perfect angle" mathematically.
Pythonpca_visualisation.py
from sklearn.decomposition import PCA
import plotly.graph_objects as go

pca = PCA(n_components=2)
X_2d = pca.fit_transform(X_scaled)

var = pca.explained_variance_ratio_
print(ff"PC1 explains {var[0]*100:.1f}% of variance")
print(ff"PC2 explains {var[1]*100:.1f}% of variance")

fig = go.Figure()

normal_mask = ~df['is_anomaly']
fig.add_trace(go.Scatter(
    x=X_2d[normal_mask, 0],
    y=X_2d[normal_mask, 1],
    mode='markers', name='Normal',
    marker=dict(color='#1A56DB', size=5, opacity=0.6)
))

anom_mask = df['is_anomaly']
fig.add_trace(go.Scatter(
    x=X_2d[anom_mask, 0],

EDA — what it is and why every project starts here

EDA (Exploratory Data Analysis) means: look at your data before you model it. You check distributions, find skewed features, spot missing values, understand correlations, identify outliers. If you skip EDA, you build models on wrong assumptions.

This project automates EDA. The skill you're building is: given an unknown dataset, what questions do you ask first?

EDA CheckWhat you look forpandas code
Missing valuesWhich columns have nulls? How many %?df.isnull().sum() / len(df) * 100
Data typesAre numbers stored as strings? Dates as objects?df.dtypes
DuplicatesAre there repeated rows?df.duplicated().sum()
DistributionNormal? Skewed? Bimodal?df.describe() + histogram
SkewnessHeavy tail? Need log transform?df.skew()
CorrelationWhich features move together?df.corr()
CardinalityHow many unique values in categorical cols?df.nunique()

KDE — making distributions look beautiful

A histogram is blocky. KDE (Kernel Density Estimation) is a smooth curve that estimates the true underlying distribution. You use scipy.stats.gaussian_kde.

Pythonkde.py
from scipy import stats
import numpy as np

data = df['revenue'].dropna()

hist_vals, bin_edges = np.histogram(data, bins=30)
bin_centers = (bin_edges[:-1] + bin_edges[1:]) / 2

kde = stats.gaussian_kde(data)

x_range = np.linspace(data.min(), data.max(), 200)
kde_y = kde(x_range)

bin_width = bin_edges[1] - bin_edges[0]
kde_y_scaled = kde_y * len(data) * bin_width

fig.add_trace(go.Bar(x=bin_centers, y=hist_vals, name='Histogram'))
fig.add_trace(go.Scatter(x=x_range, y=kde_y_scaled,
    mode='lines', line=dict(color='#DC2626', width=2), name='KDE'))

Zero-shot classification — AI without training data

Normal text classification: you train a model on thousands of labelled examples. Zero-shot: you just describe the categories in plain English and the model figures it out using its pre-trained language understanding.

Pythonzero_shot.py
from transformers import pipeline

classifier = pipeline(
    "zero-shot-classification",
    model="facebook/bart-large-mnli"
)

result = classifier(
    "The patient showed signs of high blood pressure and fatigue.",
    candidate_labels=["medical", "sports", "technology", "finance"]
)
print(result['labels'])
print(result['scores'])

result = classifier(
    "AI is being used to detect cancer earlier and more accurately.",
    candidate_labels=["healthcare", "AI", "technology", "finance"],
    multi_label=True
)

result = classifier(
    "Congratulations! You've won a prize. Click here.",
    candidate_labels=["spam", "legitimate"],
    hypothesis_template="This email is a {} message."
)

Response caching — making the API fast

ML models are slow (0.5–3 seconds per call). If someone sends the exact same text and labels twice, why run the model again? Cache the result.

Pythoncaching.py
import hashlib

_cache = {}

def cache_key(text: str, labels: list, multi_label: bool) -> str:
    data = ff"{text}|{'|'.join(sorted(labels))}|{multi_label}"
    return hashlib.md5(data.encode()).hexdigest()

async def classify(req: ClassifyRequest):
    ck = cache_key(req.text, req.labels, req.multi_label)

    if ck in _cache:
        result = _cache[ck].copy()
        result["cached"] = True
        return result

    result = run_model(req.text, req.labels, req.multi_label)
    _cache[ck] = result
    return result

The full-stack mental model

Full stack means your app has three distinct layers that each have one job:

Frontend (React) ← What the user sees and interacts with. Runs in the browser. Makes HTTP requests.
↕ HTTP (JSON over REST API)
Backend (FastAPI) ← Business logic, validation, authentication. Receives requests, queries DB, returns JSON.
↕ SQL queries
Database (SQLite) ← Stores data permanently. Tables, relationships, indexes.
Common beginner mistake: mixing these layers. Business logic doesn't belong in the database. UI logic doesn't belong in the backend. Keep each layer focused on its single responsibility.

SQLite schema design — tables, foreign keys, indexes

SQLschema.sql
CREATE TABLE IF NOT EXISTS projects (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL,
    description TEXT,
    status      TEXT DEFAULT 'active'
        CHECK(status IN ('active','paused','completed')),
    priority    TEXT DEFAULT 'medium'
        CHECK(priority IN ('low','medium','high','critical')),

CREATE TABLE IF NOT EXISTS tasks (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id) ON DELETE CASCADE,
    title      TEXT NOT NULL,

);

CREATE INDEX IF NOT EXISTS idx_tasks_project ON tasks(project_id);

FastAPI with SQLite — the dependency injection pattern

Pythondb_pattern.py
import sqlite3
from fastapi import Depends

def get_db():
    conn = sqlite3.connect("taskforge.db")
    conn.row_factory = sqlite3.Row
    conn.execute("PRAGMA foreign_keys = ON")
    try:
        yield conn
    finally:
        conn.close()

@app.get("/projects")
def list_projects(db: sqlite3.Connection = Depends(get_db)):
    rows = db.execute("SELECT * FROM projects ORDER BY created_at DESC").fetchall()
    return [dict(row) for row in rows]

@app.post("/projects", status_code=201)
def create_project(proj: ProjectCreate, db = Depends(get_db)):
    cursor = db.execute(
        """INSERT INTO projects (name, description, status)
           VALUES (?, ?, ?) RETURNING *""",
        (proj.name, proj.description, proj.status)
    )
    db.commit()
    return dict(cursor.fetchone())

CRUD operations — POST, PATCH, DELETE from React

JavaScriptcrud_operations.js
async function createTask(taskData) {
    const response = await fetch('http://localhost:8001/tasks', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(taskData)
    });
    if (!response.ok) throw new Error('Failed to create task');
    return response.json();
}

async function updateTask(taskId, changes) {
    const response = await fetch(`http://localhost:8001/tasks/${taskId}`, {
        method: 'PATCH',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(changes)
    });
    return response.json();
}

async function deleteTask(taskId) {
    await fetch(`http://localhost:8001/tasks/${taskId}`, {
        method: 'DELETE'
    });
}

async function handleStatusChange(taskId, newStatus) {
    setTasks(prev => prev.map(t =>
        t.id === taskId ? { ...t, status: newStatus } : t
    ));
    try {
        await updateTask(taskId, { status: newStatus });
    } catch(e) {
        setTasks(prev => prev.map(t =>
            t.id === taskId ? { ...t, status: 'todo' } : t
        ));
    }
}

★ How I Think When Building

The process that works behind every project. Not code — thinking.

The question I ask before writing any code

Before I open my editor, I ask: "What is the data flow?" Where does data come from? What shape is it in? What do I do to it? Where does it end up?

Every project above is just a data flow:

  • Stock Dashboard: Yahoo API → OHLCV DataFrame → compute metrics → Plotly chart → user screen
  • Anomaly Detector: User CSV → pandas → StandardScaler → IsolationForest → labels + scores → Plotly scatter → download button
  • Text Classifier: JSON request → Pydantic validates → cache check → BART model → JSON response
  • Task Manager: User click → React state update → fetch() → FastAPI → SQL → SQLite → JSON → React re-render

Draw this flow on paper before touching code. It takes 5 minutes and saves 5 hours.

The build order that always works

  1. Get the data working first

    Before any UI. Just print it. Verify the shape, types, and values are what you expect. Fix all data issues here.

  2. Build the core logic

    The computation, model, or API — in isolation, tested in a notebook or with simple print statements. No UI yet.

  3. Add the simplest possible UI

    Get something visible. Ugly is fine. The point is seeing real output from your real logic.

  4. Handle errors and edge cases

    What if the data is empty? What if the API fails? What if someone uploads the wrong file type? Handle these explicitly.

  5. Polish and deploy

    Clean up the UI, write a README, push to GitHub, deploy to Streamlit Cloud / Railway. Get a live URL.

The most important thing I can tell you: Build things that work end-to-end, even if ugly, before making anything beautiful. A working ugly project beats a beautiful project that crashes on real data every single time. Clients and employers care about working, not perfect.
